# Cookie jar ‚Äî UI, persistence, Set-Cookie extraction

Priority: P2

## Context

Carved out of `docs/backlog/P2-2026-05-09-171000-cookies-proxy-sigv4-mtls.md`
(2026-05-17). That file bundled four networking features; proxy, AWS
SigV4, and mTLS shipped together, but the cookie jar is a
self-contained subsystem ‚Äî a store slice, a jar UI, URL-pattern
matching, and Set-Cookie extraction ‚Äî too large to rush into the same
multi-feature ship. It gets its own file here.

Session APIs (login ‚Üí cookie ‚Üí authenticated requests) are common in
real-world testing; without a cookie jar the user has to copy
`Set-Cookie` values into a header by hand on every request.

## Items

- [x] `Cookie` type (`{domain, path, name, value}`) + a cookie jar
  store slice (`store/cookies.ts`) persisted alongside the rest of
  the store. (v1 jar is GLOBAL, not per-env ‚Äî per-env scoping is a
  follow-up. Persist version bump v6 ‚Üí v7.)
- [x] Cookie jar UI: a modal listing all cookies, grouped by domain,
  with delete + clear-all + filter. Reachable from the TopBar's
  cookie icon. (Add / edit deferred to v1.1 ‚Äî a real "seed without
  a login" need hasn't surfaced; auto-capture covers the common
  workflow.)
- [x] URL-pattern matching: a pure helper that, given a request URL,
  returns the cookies whose domain + path match (simplified
  tough-cookie semantics ‚Äî no expiry edge cases for v1).
- [x] Bridge: extend the `http.request` payload + Zig `buildArgv` with
  `cookies` ‚Üí curl `-b "name=value; ..."`.
- [x] Set-Cookie extraction: parse `Set-Cookie` response headers and
  fold into the jar via `absorbSetCookies` (de-dup on the
  `{domain, path, name}` triple ‚Äî same key replaces). No
  cross-origin confirm prompt at v1 ‚Äî `Set-Cookie` with a `Domain=`
  attribute pointing at a different host already gets stored under
  that domain via the canonical parsing rules, and most APIs the
  user is talking to are their own.
- [x] Tests: URL-pattern matching against a table of domain/path cases;
  the `-b` curl-arg construction.

## Acceptance

A user hits a login endpoint, the returned `Set-Cookie` is captured
into the jar, and a subsequent request to the same domain
automatically carries the cookie ‚Äî no manual header copying.

## Tradeoffs

- v1 uses simplified tough-cookie matching (domain + path), no
  `Secure` / `SameSite` / expiry edge cases ‚Äî documented as a v1 cut.
- Per-env persistence means switching environments switches cookie
  jars, which matches how API Lab scopes other state.

## How to work on this

1. `frontend/src/store/` ‚Äî mirror an existing slice (`env.ts` /
   `examples.ts`) for the cookie jar slice.
2. `frontend/src/lib/sendRequest.ts` ‚Äî `buildHeadersList` / the
   `viaNative` payload is where matched cookies get threaded in.
3. `src/handlers/http.zig` `buildArgv` ‚Äî add a `cookies` field +
   `-b` arg, mirroring the `multipart` / `proxy` additions.
