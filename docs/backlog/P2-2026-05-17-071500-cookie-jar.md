# Cookie jar — UI, persistence, Set-Cookie extraction

Priority: P2

## Context

Carved out of `docs/backlog/P2-2026-05-09-171000-cookies-proxy-sigv4-mtls.md`
(2026-05-17). That file bundled four networking features; proxy, AWS
SigV4, and mTLS shipped together, but the cookie jar is a
self-contained subsystem — a store slice, a jar UI, URL-pattern
matching, and Set-Cookie extraction — too large to rush into the same
multi-feature ship. It gets its own file here.

Session APIs (login → cookie → authenticated requests) are common in
real-world testing; without a cookie jar the user has to copy
`Set-Cookie` values into a header by hand on every request.

## Items

- [ ] `Cookie` type (`{domain, path, name, value}`) + a per-env cookie
  jar store slice (`store/cookies.ts`) persisted alongside the rest of
  the store.
- [ ] Cookie jar UI: a modal listing all cookies, grouped/scoped by
  domain, with add / edit / delete. Reachable from the TopBar or
  Settings.
- [ ] URL-pattern matching: a pure helper that, given a request URL,
  returns the cookies whose domain + path match (simplified
  tough-cookie semantics — no expiry edge cases for v1).
- [ ] Bridge: extend the `http.request` payload + Zig `buildArgv` with
  `cookies` → curl `-b "name=value; ..."`.
- [ ] Set-Cookie extraction: parse `Set-Cookie` response headers and
  offer to add them to the jar, with a confirm prompt for cookies
  whose domain differs from the request host (cross-origin).
- [ ] Tests: URL-pattern matching against a table of domain/path cases;
  the `-b` curl-arg construction.

## Acceptance

A user hits a login endpoint, the returned `Set-Cookie` is captured
into the jar, and a subsequent request to the same domain
automatically carries the cookie — no manual header copying.

## Tradeoffs

- v1 uses simplified tough-cookie matching (domain + path), no
  `Secure` / `SameSite` / expiry edge cases — documented as a v1 cut.
- Per-env persistence means switching environments switches cookie
  jars, which matches how API Lab scopes other state.

## How to work on this

1. `frontend/src/store/` — mirror an existing slice (`env.ts` /
   `examples.ts`) for the cookie jar slice.
2. `frontend/src/lib/sendRequest.ts` — `buildHeadersList` / the
   `viaNative` payload is where matched cookies get threaded in.
3. `src/handlers/http.zig` `buildArgv` — add a `cookies` field +
   `-b` arg, mirroring the `multipart` / `proxy` additions.
