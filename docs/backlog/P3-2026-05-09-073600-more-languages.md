## Context

Originally captured via /inbox on 2026-05-09 07:36:00.

The i18n system is in place with Turkish + English. `tr.ts` is the
source of truth — TypeScript fails the build if another locale is
missing a key, so adding a language is mechanical and the type
checker guarantees completeness. Per-language steps: create
`frontend/src/lib/i18n/<code>.ts` exporting a `Dict`, register it in
`index.ts`'s `locales` map + `LOCALE_LABEL`, append to
`SUPPORTED_LOCALES`, and add the `lang.<code>` label key to every
existing locale.

## Items

- [ ] **Spanish (ES), German (DE), French (FR).** Latin-script locales —
  no layout work, just translation. One slice each (or one combined
  slice). Touchpoints: `frontend/src/lib/i18n/`.
- [ ] **Japanese (JA), Simplified Chinese (ZH).** CJK locales — translate
  and sanity-check that no fixed-width UI clips longer/CJK glyphs.
  Touchpoints: `frontend/src/lib/i18n/`, spot-check component widths.
- [ ] **Arabic (AR) + RTL audit.** Translate, then audit layout for
  RTL: `dir="rtl"` on `<html>` when an RTL locale is active, mirror
  flex/grid where directional, and verify Tailwind logical properties
  (start/end) are used instead of left/right. Touchpoints:
  `frontend/src/lib/i18n/`, `App.tsx` / layout components, `main.css`.

## Acceptance

Each new locale is selectable, every UI string renders translated
(TypeScript guarantees no missing keys), and Arabic renders correctly
right-to-left with no mirrored-layout glitches.

## Tradeoffs & risks

- Translation quality — machine translation gets the build green but
  not necessarily idiomatic; flag locales that want a human pass.
- AR is the only structurally risky one (RTL); the Latin/CJK locales
  are pure data.
- Each locale adds a small bundle increment (one `Dict` object).

## How to work on this

Ship the Latin-script trio first (zero layout risk), then CJK, then
Arabic last since it carries the RTL audit. For each: copy `en.ts`,
translate values, register in `index.ts`, extend `SUPPORTED_LOCALES`,
add the `lang.<code>` key everywhere, and let `dnpm run typecheck`
prove completeness. For AR, do the RTL layout audit as a distinct
step before marking it done.
